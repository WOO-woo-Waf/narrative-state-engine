# 作者工作台前端执行方案：模型辅助审计与批量动作

本文档是前端窗口专用执行方案，目标是在作者工作台中实现“像 GPT 一样对话审计状态候选”的体验，同时提供勾选批量处理能力。

前端要提供的核心能力：

```text
候选多选
批量接受/拒绝
风险分组
模型审计助手
审计草案卡片
草案详情编辑
草案执行确认
执行结果展示
状态刷新
全中文界面
```

一句话目标：

```text
作者不必逐条看完 85 个候选，可以让模型生成多份审计草案，作者确认后前端提交执行并展示结果。
```

## 一、前端目标

候选审计页要从“表格工具”升级为“审计工作区”。

作者应该可以：

```text
看候选统计
看风险分组
一键选择低风险候选
批量接受或拒绝
和模型对话说明审计策略
查看模型生成的多份草案
修改草案
确认执行草案
查看执行结果
刷新状态环境和迁移图
```

## 二、页面结构

建议在工作台左侧菜单中新增或强化：

```text
候选审计
模型审计
草案记录
```

或者在“候选审计”页面内使用标签页：

```text
模型审计
候选列表
草案记录
执行结果
```

默认进入“模型审计”，不要让作者一上来面对 85 条候选。

推荐布局：

```text
顶部：当前小说、当前任务、候选统计、风险分布、刷新按钮
中间：模型审计对话
右侧：草案卡片和当前上下文摘要
下方或独立标签：候选列表
```

所有中文为主语言。

## 三、候选列表批量选择

候选列表必须支持：

```text
全选当前页
全选当前筛选结果
取消选择
反选
仅选择低风险
仅选择中风险
仅选择高风险
仅选择缺证据项
```

批量按钮：

```text
批量接受
批量拒绝
标记冲突
保留待审计
生成审计草案
交给模型审计
```

按钮显示数量：

```text
批量接受（12）
批量拒绝（3）
生成草案（18）
```

执行前必须显示中文预览：

```text
本次将接受 12 个候选。
其中低风险 10 个，中风险 2 个。
预计产生 12 条状态迁移。
请输入“确认执行”继续。
```

## 四、风险分组视图

候选审计页新增风险筛选：

```text
全部
低风险
中风险
高风险
极高风险
冲突
缺证据
需要作者确认
```

每个候选行显示：

```text
风险等级
推荐动作
风险原因摘要
```

详情区显示：

```text
完整风险原因
推荐处理方式
是否需要作者确认
是否被作者锁定保护
是否涉及参考文本覆盖主状态
```

风险颜色建议：

| 风险 | 显示 |
| --- | --- |
| 低风险 | 绿色 |
| 中风险 | 黄色 |
| 高风险 | 橙色 |
| 极高风险 | 红色 |

## 五、模型审计助手

新增主功能：

```text
模型审计助手
```

体验参考 GPT 对话：

```text
消息流
输入框
发送按钮
上下文摘要
模型生成的草案卡片
```

输入框占位文本：

```text
告诉模型你想怎么审计。例如：低风险设定先通过，人物关系保留待确认。
```

常用快捷指令：

```text
帮我保守审计
只处理低风险候选
列出高风险问题
生成三份审计草案
解释当前冲突
把人物关系全部保留待确认
```

发送后展示：

```text
模型回复
草案卡片
需要我确认的问题
高风险提醒
```

## 六、审计上下文摘要

模型审计助手顶部必须展示压缩后的上下文，不显示完整 85 条。

示例：

```text
当前任务：真实 123 主故事分析审计
候选总数：85
待审计：83
已接受：2

风险分布：
低风险：24
中风险：31
高风险：25
极高风险：5

建议优先：
1. 先接受低风险地点、术语、世界规则
2. 人物整卡保留待人工确认
3. 关系和剧情线列出冲突点
4. 证据不足项先不入库
```

如果后端暂时没有风险分布，前端显示：

```text
风险评估尚未生成，请点击“生成风险评估”。
```

## 七、草案卡片

模型生成的每份草案显示为卡片。

卡片内容：

```text
草案名称
风险等级
草案摘要
将接受多少项
将拒绝多少项
将标记冲突多少项
将保留多少项
预计新增/更新多少状态对象
需要确认的问题
```

按钮：

```text
查看详情
执行草案
复制为新草案
取消草案
让模型修改草案
```

执行前确认：

```text
请输入“确认执行”以执行该审计草案。
```

高风险草案确认：

```text
请输入“确认高风险写入”以继续。
```

## 八、草案详情与人工调整

草案详情可以是抽屉、弹窗或独立页。

显示字段：

```text
动作项列表
候选编号
候选摘要
动作类型
风险等级
模型理由
预期影响
是否会写入状态
是否会产生迁移
```

作者可以调整：

```text
删除某个动作项
把接受改成保留
把拒绝改成冲突
给某项加备注
重新计算预览
```

调整后必须保存为新草案版本或更新草案。

## 九、执行结果视图

执行完成后显示清晰结果：

```text
执行完成
已接受：10
已拒绝：2
已标记冲突：1
已跳过：3
失败：0
动作编号：action-001
迁移编号：transition-001, transition-002
更新对象：state-object-001
```

失败项要可展开：

```text
候选编号
失败原因
阻塞原因
建议下一步
```

执行结果不能只放 toast，必须在“执行结果”区域长期可见。

## 十、状态刷新

执行批量操作或草案后，前端必须刷新：

```text
候选列表
状态环境
状态对象
迁移图
任务列表
执行结果
```

如果后端返回：

```text
environment_refresh_required = true
graph_refresh_required = true
```

前端必须自动触发刷新。

## 十一、中文化要求

所有主界面文字为中文。

按钮：

```text
发送
刷新状态
生成草案
执行草案
取消草案
批量接受
批量拒绝
标记冲突
保留待审计
查看详情
让模型修改
确认执行
```

状态：

```text
待审计
已接受
已拒绝
有冲突
已跳过
执行中
执行完成
执行失败
```

错误：

```text
模型审计失败
草案创建失败
草案执行失败
候选刷新失败
状态刷新失败
后端返回成功，但没有写入任何状态
```

技术字段可保留，但要有中文解释：

```text
动作编号：action_id
迁移编号：transition_id
候选编号：candidate_item_id
草案编号：draft_id
```

## 十二、前端接口对接

需要对接后端接口：

```text
获取候选列表
获取风险评估
批量审计候选
创建审计草案
获取审计草案
确认审计草案
执行审计草案
取消审计草案
发送模型审计消息
获取任务状态
获取状态环境
获取迁移图
```

前端 API client 要做兼容：

```text
后端字段缺失时显示“暂无”
数组为空时显示中文空态
接口失败时显示中文错误
执行中任务显示进度
```

## 十三、测试计划

前端必须新增测试：

1. 候选多选测试。
2. 全选当前页测试。
3. 仅选择低风险测试。
4. 批量操作确认弹窗测试。
5. 模型审计消息发送测试。
6. 草案卡片渲染测试。
7. 草案详情调整测试。
8. 草案执行结果展示测试。
9. 执行后刷新候选和图谱测试。
10. 中文关键文案测试。

运行：

```powershell
cd web/frontend
rtk npm run typecheck
rtk npm test
rtk npm run build
rtk npm run e2e -- --reporter=line
```

## 十四、真实数据验收

使用：

```text
小说编号：story_123_series_realrun_20260510
任务编号：task_123_series_realrun_20260510
候选数量：约 85
```

最低通过：

1. 能看到候选总数和风险分布。
2. 能一键选择低风险候选。
3. 能批量接受候选。
4. 能批量拒绝候选。
5. 能打开模型审计助手。
6. 能发送审计策略消息。
7. 能看到模型生成的草案卡片。
8. 能查看草案详情。
9. 能执行草案。
10. 能看到执行结果。
11. 执行后候选状态刷新。
12. 执行后迁移图刷新。
13. 全界面中文可读。

推荐通过：

1. 草案可人工调整。
2. 执行中显示进度。
3. 失败项可展开查看。
4. 模型能生成多份不同策略草案。
5. 高风险草案要求更强确认。
