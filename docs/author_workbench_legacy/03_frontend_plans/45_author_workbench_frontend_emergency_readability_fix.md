## 二、最高优先级问题

### 问题编号：前端紧急-001

问题名称：全站主语言必须是中文

严重程度：阻塞

用户要求：

```text
所有内容都要汉化。
不要把主界面换成英文。
中文是主语言。
所有状态、按钮、标题、提示、空态、错误说明都要优先中文显示。
```

修复要求：

1. 页面标题、按钮、标签、筛选器、状态栏、提示语全部改成中文。
2. 不允许出现乱码中文。
3. 不允许把乱码中文临时改成英文了事。
4. 技术编号可以保留原样，例如 `story_id`、`task_id`、`candidate_item_id`，但必须有中文标签：
   - 小说编号：`story_id`
   - 任务编号：`task_id`
   - 候选编号：`candidate_item_id`
5. 状态值要显示中文解释：
   - `pending_review` 显示为“待审计”
   - `accepted` 显示为“已接受”
   - `rejected` 显示为“已拒绝”
   - `conflicted` 显示为“有冲突”
   - `canonical` 显示为“主状态”
   - `candidate` 显示为“候选”
6. 来源角色要显示中文：
   - `primary_story` 显示为“主故事”
   - `same_world_reference` 显示为“同世界观参考”
   - `crossover_reference` 显示为“联动参考”
   - `style_reference` 显示为“风格参考”
7. 数据库状态要显示中文：
   - 在线：数据库在线
   - 离线：数据库离线
   - 加载中：正在检查数据库
   - 失败：数据库检查失败

验收标准：

```text
作者不懂前端技术，也能看懂当前页面在展示什么、按钮会做什么、失败原因是什么。
```

### 问题编号：前端紧急-002

问题名称：当前三栏布局不适合真实状态审计

严重程度：阻塞

用户反馈：

```text
现在的三栏页面布局非常有问题。
无法看完整相关内容。
状态审计下面没有分页。
页面被拉得无限长。
候选审计只读检查可视化非常差。
```

当前问题：

1. 三栏同时存在时，中间主体宽度不足。
2. 候选表、状态环境、图谱、对话互相抢空间。
3. 右侧图谱和检查区太窄，无法阅读。
4. 候选审计区没有分页，真实数据 85 条候选会把页面拉得很长。
5. 状态审计内容没有分层，作者无法确定自己正在看哪个任务、哪个候选、哪个字段。

结论：

```text
当前三栏布局不适合作为默认工作台布局。
必须改为“任务为中心”的主界面。
```

## 三、主界面重构方案

### 三点一、设计原则

作者工作台不是仪表盘堆叠页，而是任务工作流页面。

主线应该围绕任务推进：

```text
小说
  -> 任务
  -> 状态分析
  -> 候选审计
  -> 状态维护
  -> 剧情规划
  -> 续写任务
  -> 分支审稿
  -> 状态入库或丢弃
```

前端布局要服务这个流程。

### 三点二、推荐主布局

默认改为两层结构：

```text
顶部：当前小说、当前任务、数据库状态、任务状态、刷新按钮

左侧：任务菜单
  - 任务总览
  - 状态分析
  - 候选审计
  - 状态维护
  - 剧情规划
  - 续写任务
  - 分支审稿
  - 图谱
  - 设置

中间：当前任务主体区域

右侧：可选详情抽屉
  - 当前候选详情
  - 当前状态对象详情
  - 当前动作详情
  - 当前证据详情
```

不要默认三栏全展开。

右侧详情抽屉应满足：

1. 默认可以关闭。
2. 打开时不挤压主体到不可读。
3. 可以应用内全屏。
4. 可以固定宽度或拖拽宽度。

### 三点三、图谱处理方式

图谱不要默认塞进三栏右侧。

可选两种方案，任选其一：

方案一：独立菜单页。

```text
/workbench-v2/graph
```

或在同一工作台内切换到“图谱”菜单后，主区域全宽显示图。

方案二：详情抽屉中的小预览 + 独立大图。

```text
候选审计页：只显示小型关系摘要
图谱页：全屏显示状态图、迁移图、分析图、分支图
```

验收要求：

```text
图谱必须能大面积显示。
全屏/展开后不被侧栏、顶部栏、详情栏遮挡。
点击节点后详情清楚显示。
```

## 四、候选审计页重构

### 四点一、候选审计必须分页

当前真实任务有 85 个候选项，不分页会把页面无限拉长。

必须增加：

```text
分页
每页数量选择
总数显示
当前页显示
上一页
下一页
跳转页码
```

默认建议：

```text
每页 20 条
可选 10 / 20 / 50 / 100
```

页面顶部显示：

```text
候选总数：85
待审计：83
已接受：2
已拒绝：0
有冲突：0
当前显示：第 1 页，每页 20 条
```

### 四点二、候选审计主体布局

候选审计页建议改为：

```text
上方：筛选和统计
中间：候选列表
右侧或下方：候选详情
底部：分页
```

候选列表只放摘要字段：

| 中文列名 | 内容 |
| --- | --- |
| 状态 | 待审计/已接受/已拒绝/有冲突 |
| 类型 | 角色/关系/场景/世界规则等 |
| 名称 | 对象名称或候选摘要 |
| 字段 | 字段路径 |
| 操作 | 新增/更新/删除/合并 |
| 置信度 | 数值和颜色 |
| 来源 | 主故事/参考文本 |
| 风险 | 低/中/高 |

不要在表格里直接铺完整 JSON。

### 四点三、候选详情区

点击候选后，详情区展示完整内容：

```text
候选基础信息
  候选编号
  候选集合
  目标对象
  对象类型
  字段路径
  操作类型
  当前状态
  来源角色
  权威等级
  置信度

变更对比
  修改前
  修改后
  差异摘要

证据
  证据编号
  来源文档
  相关片段
  证据类型

风险提示
  缺失字段
  冲突原因
  阻塞问题

操作结果
  是否写入
  动作编号
  迁移编号
  更新对象
  警告
```

复杂对象展示规则：

1. 默认显示摘要。
2. 点击“展开完整内容”后显示格式化数据。
3. 长文本区域固定高度，可滚动。
4. 不允许内容覆盖按钮。
5. 不允许候选详情撑爆整个页面。

### 四点四、审计操作区

接受、拒绝、锁定、标记冲突等按钮放在详情区底部，不要塞进表格列里。

按钮必须中文：

```text
接受候选
拒绝候选
标记冲突
锁定字段
查看证据
刷新候选
```

高风险操作要有中文确认：

```text
请输入“确认接受”
请输入“确认锁定”
请输入“确认入库”
```

不要要求作者输入英文确认词。

### 四点五、筛选器

候选审计页至少需要这些筛选：

```text
状态
类型
来源
权威等级
置信度
是否有证据
是否有冲突
是否已锁定
关键词搜索
```

筛选器本身也必须中文。

## 六、图谱页重构

图谱应该成为独立工作区。

推荐菜单名：

```text
图谱
```

图谱页包含：

```text
状态图
迁移图
分析图
分支图
```

切换方式：

```text
顶部标签页
```

基础功能：

1. 适配画布。
2. 居中显示。
3. 放大缩小。
4. 拖拽节点。
5. 点击节点打开详情抽屉。
6. 筛选节点类型。
7. 显示空图原因。
8. 显示接口降级原因。
9. 应用内全屏。

当前真实数据期望：

```text
状态图：至少 2 个已确认状态对象节点
迁移图：至少 2 条候选接受迁移，且有动作编号
```

如果图谱页看不到这些内容，就是前端渲染或接口路径问题。

## 七、顶部状态栏重构

顶部状态栏必须服务作者判断当前是不是“可以操作”。

推荐显示：

```text
当前小说
当前任务
当前场景
数据库状态
状态版本
待审计候选
运行中任务
当前分支
刷新状态
```

所有标签中文。

数据库状态刷新要求：

1. 页面打开时请求健康接口。
2. 点击“刷新状态”时刷新健康、小说、任务、环境、候选、任务队列。
3. 页面重新获得焦点时可轻量刷新健康状态。
4. 如果健康接口显示数据库在线，但环境接口失败，要显示：

```text
数据库在线，但当前任务环境加载失败。
```

不能继续显示笼统的“数据库离线”。

## 八、任务为中心的页面组织

左侧菜单以任务为中心，建议结构如下：

```text
任务总览
  显示当前任务状态、来源文档、候选统计、最近动作

状态分析
  显示分析结果、分析覆盖率、来源文档、证据概览

候选审计
  审核分析产生的状态候选

状态维护
  查看和修改主状态对象

剧情规划
  作者和模型协作确定后续状态转移计划

续写任务
  根据状态和规划创建续写任务

分支审稿
  审核生成草稿，决定接受、重写、丢弃

图谱
  查看状态图、迁移图、分析图、分支图

设置
  上下文预算、并发、模型、检索范围等配置
```

这个结构比三栏同时显示更符合实际工作流。
