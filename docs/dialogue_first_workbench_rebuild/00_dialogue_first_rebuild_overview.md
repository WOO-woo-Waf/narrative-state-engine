# 对话主入口作者工作台总体方案

## 一、重大改动点

作者工作台的主界面要从“多面板管理后台”重构为“对话式小说状态环境”。

这次重构不是推翻之前的状态机系统。恰恰相反，**状态仍然是一等公民，状态机仍然是系统核心**。对话主入口只是更适合作者使用的一层操作界面。

必须保留的底层核心：

```text
小说状态环境
状态对象
状态候选
状态迁移
作者锁定字段
证据索引
任务记录
分支记录
图谱投影
```

分析、审计、规划、续写、分支审稿都不是互相割裂的新系统，而是围绕同一套状态机做状态产生、状态确认、状态转移、状态展示和状态回流。

旧范式：

```text
左侧选择任务
中间看表格
右侧看状态/图谱
用户点按钮执行
```

新范式：

```text
作者在对话线程中提出意图
模型读取当前任务上下文
模型生成动作草案
作者确认草案
系统执行任务
执行结果回到对话线程
状态环境同步更新
```

对话线程类似 ChatGPT 或 Codex：

```text
中间是对话和运行过程
底部是输入框
运行过程以卡片、日志、草案、结果的形式插入对话
```

## 二、主界面不是候选审计表

候选审计表、状态图、状态对象、证据检索都不是主界面。它们是对话环境中的辅助工具。

这里的“辅助”不是删除或弱化。它们必须保留，并且要比之前展示得更好：

```text
候选审计表用于字段级人工检查。
状态对象页用于查看角色卡、地点、世界规则、剧情线。
图谱页用于展示小说状态发展、人物关系、状态迁移、分支演化。
证据页用于追溯原文、参考文本、风格样例和检索结果。
```

新的对话主入口负责把作者带到正确的上下文，并生成可执行动作；图谱和表格负责把状态机的结构清楚展示出来。

主界面应该是：

```text
对话线程
  + 当前任务上下文
  + 模型动作草案
  + 执行过程
  + 执行结果
```

作者可以说：

```text
请分析这份主故事，并把 2/3 只作为参考证据。
请帮我审计当前候选，低风险设定先生成通过草案。
请把角色关系中有冲突的地方列出来，不要写入。
请根据我刚才确认的规划生成下一章草稿分支。
请对这个分支做一致性检查，并生成修订建议。
```

系统把这些话转成：

```text
任务草案
审计草案
状态修改草案
续写任务草案
分支审稿草案
```

作者点击执行后，后端再真正运行。

## 三、任务与上下文

小说下面有多个任务。

任务类型：

```text
分析任务
从零创建状态任务
候选审计任务
状态维护任务
剧情规划任务
续写任务
分支审稿任务
修订任务
```

每个任务都有自己的上下文场景。

上下文场景不是新状态系统，而是同一套小说状态环境的不同视图：

```text
分析上下文
  原文、章节块、证据、分析 schema、候选生成规则。

审计上下文
  候选集合、风险分布、证据、当前主状态、作者锁定字段。

状态维护上下文
  已确认状态对象、字段历史、作者意图、修改草案。

剧情规划上下文
  当前主状态、伏笔、剧情线、作者目标、禁止破坏项。

续写上下文
  主状态、规划、风格样例、检索证据、生成参数。

分支审稿上下文
  草稿正文、状态差异、分支基线、入库风险。
```

因此，后续代码落地必须遵守：

```text
不要新建一套与 NovelState/StateEnvironment 平行的状态系统。
不要让对话线程自己保存一份不可追踪的小说状态。
不要把分析状态、审计状态、续写状态做成互相割裂的数据孤岛。
```

对话线程可以保存消息、动作草案和执行结果，但真正的小说信息仍回流到统一状态环境。

## 四、模型是环境中的执行合作者

模型在当前上下文中不是自由发挥，而是通过工具和协议工作。

模型可以做：

```text
读取上下文
提出分析计划
生成动作草案
解释风险
比较草案
请求作者确认
总结执行结果
继续下一轮操作
```

模型不可以做：

```text
未经确认直接写入主状态
绕过作者锁定字段
让参考文本覆盖主故事状态
把执行失败说成成功
隐藏冲突或跳过项
```

这个系统更像“React 状态 + 工具调用”的组合：

```text
状态机提供当前环境。
上下文构建器把状态机切成适合模型阅读的上下文。
模型根据上下文生成动作草案。
前端把动作草案展示为按钮和卡片。
作者确认后，后端工具执行。
工具执行结果再次回写状态机。
```

模型可以自由提出要调用哪些工具，但不能绕过确认协议直接执行有风险的写入。

## 四点五、对话压缩与上下文切换

对话可以压缩，但压缩对象是“对话历史”，不是小说状态。

```text
可以压缩：
  旧对话消息
  重复解释
  已完成任务过程
  被新意图覆盖的旧意图

不可以压缩掉：
  已确认角色状态
  已确认世界规则
  作者锁定字段
  状态迁移历史
  分支记录
  证据索引
```

上下文可以切换，但核心状态仍然一致：

```text
分析上下文 -> 看原文、章节、证据、候选生成规则。
审计上下文 -> 看候选、风险、证据、主状态。
规划上下文 -> 看主状态、伏笔、剧情线、作者目标。
续写上下文 -> 看主状态、规划、风格、检索证据。
```

这些上下文像不同工作台视角。它们使用同一本小说的状态环境，不是不同的小说状态副本。

作者也可以切换到一个新线程或新上下文：

```text
旧线程保留。
旧上下文保留。
新线程重新构建上下文。
核心状态仍指向同一个 story_id 的状态环境。
```

## 五、动作草案协议

所有会改变系统状态的行为都先生成动作草案。

动作草案类型：

```text
分析任务草案
候选审计草案
状态修改草案
剧情规划草案
续写任务草案
分支入库草案
修订草案
```

草案内容：

```text
草案标题
草案摘要
目标任务
将执行的工具
输入参数
影响范围
风险等级
需要确认的问题
执行前预览
```

作者确认后，系统才执行。

对话中的功能按键本质上也是动作草案的展示：

```text
确认执行分析
确认执行审计
确认写入状态
确认锁定字段
确认创建续写任务
确认接受分支
```

按钮不能直接越权写入。点击后仍需走后端确认策略、状态版本校验和权限校验。

## 六、运行过程展示

执行任务时，对话线程里要出现类似 Codex 的运行过程：

```text
正在读取小说状态环境
正在检索证据
正在生成候选风险评估
已生成 3 份审计草案
等待作者确认
正在执行草案 A
已接受 12 项，跳过 2 项，产生 12 条迁移
```

这些不是弹窗，而是对话线程里的运行卡片。

## 七、辅助面板

主界面仍然需要辅助面板，但它们不抢主位。

建议：

```text
左侧：小说与任务列表
中间：对话线程和输入框
右侧：可折叠上下文面板
底部或弹出：草案详情、证据、图谱、状态对象
```

辅助面板包括：

```text
当前上下文摘要
候选风险分布
动作草案列表
状态对象详情
证据详情
图谱入口
任务运行队列
分支列表
```

图谱可以独立页面或全屏工作区，不要塞进默认三栏。

图谱展示要继续保留并增强。它是状态机最重要的可视化方式之一。

图谱至少包括：

```text
状态对象图：角色、地点、组织、物件、世界规则、剧情线。
关系图：人物关系、组织归属、物件持有、地点关联。
迁移图：候选接受、字段锁定、状态修改、分支入库。
分支图：不同续写草稿、主线入库、分支 fork、重写路线。
分析图：原文块、证据、候选、状态对象之间的来源链。
```

对话主入口可以在消息中引用图谱，例如：

```text
已生成 12 条状态迁移，点击查看迁移图。
这个候选会影响 3 个角色关系，点击查看关系图。
该续写分支与主线版本存在漂移，点击查看分支图。
```

但图谱本身应该在独立大屏或独立工作区中展示。

## 八、第一阶段范围

第一阶段只落地两个场景：

```text
分析任务
候选审计任务
```

原因：

1. 分析是状态产生入口。
2. 审计是状态入库入口。
3. 这两个场景跑通后，规划和续写可以复用同一套对话、草案、执行框架。

第一阶段必须做到：

```text
作者通过对话创建分析任务草案
作者确认后执行分析
分析结果进入候选集合
作者切换审计上下文
模型根据候选生成多份审计草案
作者确认后执行审计草案
执行结果进入状态环境
对话线程展示全过程
```

## 九、最终目标

这个系统的核心不是前端表格，也不是单次生成。

核心是：

```text
围绕某本小说的状态环境，作者与模型持续对话，模型通过工具维护状态、规划状态转移、执行续写，并把每一次变化都变成可审计的动作。
```

作者工作台要成为一个写小说的人驾驭模型的环境，而不是一个堆接口的后台。

补充一句：旧代码中已经实现的状态机、候选审计、图谱投影、RAG、分支、任务和数据库能力都应保留。新工作台是在这些能力之上换一个更自然、更强的作者操作层，而不是把已有能力清空重写。
