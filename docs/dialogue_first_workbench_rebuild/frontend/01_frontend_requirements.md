# 前端需求：对话主入口作者工作台

本文档给前端窗口使用。目标是把作者工作台重构成类似 ChatGPT/Codex 的对话式工作环境。

重要约束：

```text
对话主入口不是删除旧功能。
状态机仍然是系统核心。
候选审计、状态对象、图谱、证据、分支等页面都要保留为辅助工作区。
这次是换一个更强的主操作方式，而不是把已有能力清空重写。
```

## 一、主界面原则

主界面不是表格，不是三栏后台，不是图谱页。

主界面是：

```text
对话线程
  + 运行卡片
  + 草案卡片
  + 结果卡片
  + 底部输入框
```

作者的主要动作来自输入框：

```text
请分析这些输入。
请帮我审计候选。
请生成三份审计草案。
请规划下一章。
请根据草案续写。
```

按钮和表格是辅助，不是主入口。

这里的“辅助”含义是：

```text
默认入口是对话。
需要细看时，可以打开候选表。
需要理解结构时，可以打开图谱。
需要追证据时，可以打开证据面板。
需要查看对象时，可以打开状态对象详情。
```

这些能力必须保留，并且要从对话消息、草案卡片、运行结果中一键跳转。

## 二、推荐布局

```text
左侧：小说与任务栏
中间：对话线程和输入框
右侧：上下文/草案/证据/状态详情抽屉
顶部：当前小说、当前任务、当前场景、数据库状态
```

左侧：

```text
小说列表
任务列表
新建任务
任务场景切换
```

中间：

```text
消息流
运行过程
动作草案
任务结果
底部输入框
```

右侧：

```text
当前上下文摘要
候选风险分布
草案列表
选中对象详情
证据详情
图谱入口
```

右侧必须可关闭、可展开、可全屏。

## 三、对话线程组件

消息类型：

```text
用户消息
助手回复
系统事件
工具调用
工具结果
动作草案
运行状态
结果 artifact
错误消息
```

每种消息要有清晰中文标题。

示例：

```text
正在构建审计上下文
已读取 85 个候选
已生成 3 份审计草案
等待作者确认
正在执行草案 A
执行完成：接受 12 项，跳过 2 项
```

## 四、底部输入框

输入框必须像主工作入口。

功能：

```text
多行输入
发送
停止生成
上传/选择文件
选择当前场景
插入快捷指令
引用候选/状态对象
```

快捷指令：

```text
/分析
/审计
/状态
/规划
/续写
/分支
/证据
```

中文占位：

```text
告诉模型你想做什么。例如：帮我审计当前候选，低风险设定先生成通过草案。
```

## 五、任务场景切换

场景切换不应该离开对话范式。

场景：

```text
分析
审计
状态维护
剧情规划
续写
分支审稿
修订
```

切换后：

```text
对话线程仍在。
上下文摘要更新。
可用工具更新。
输入框提示更新。
历史消息保留。
```

## 六、动作草案卡片

模型生成的草案要在对话线程中显示为卡片。

草案卡片字段：

```text
草案标题
草案摘要
风险等级
影响范围
将执行的动作
需要确认的问题
预计结果
```

按钮：

```text
查看详情
执行
取消
让模型修改
复制为新草案
```

执行前中文确认：

```text
请输入“确认执行”
```

高风险：

```text
请输入“确认高风险写入”
```

## 七、运行卡片

运行卡片展示类似 Codex 的执行过程。

状态：

```text
排队中
运行中
等待确认
已完成
已失败
已取消
```

内容：

```text
当前步骤
进度
工具名称
输入摘要
输出摘要
错误原因
下一步建议
```

运行卡片应可展开，显示详细日志。

## 八、结果 artifact

任务执行结果不要只弹 toast。

结果以 artifact 卡片显示：

```text
分析结果摘要
候选集合摘要
审计执行结果
状态变更摘要
续写草稿分支
分支审稿报告
```

artifact 可以打开详情页。

## 九、辅助工作区

候选表、状态环境、图谱仍然保留，但变成辅助工作区。

入口：

```text
查看候选
查看状态
查看图谱
查看证据
查看任务
```

要求：

1. 不要默认三栏挤在一起。
2. 图谱用独立大屏或页面。
3. 候选表分页。
4. 详情抽屉可关闭。
5. 主对话始终是中心。

辅助工作区必须支持从对话跳转：

```text
点击候选编号 -> 打开候选详情
点击状态对象 -> 打开角色卡/地点/规则详情
点击迁移编号 -> 打开迁移图并高亮
点击分支编号 -> 打开分支图或分支审稿
点击证据编号 -> 打开证据详情
```

图谱工作区要更强，而不是更弱：

```text
状态对象图
人物关系图
状态迁移图
分支演化图
分析证据图
```

图谱应独立大屏展示，不再挤在默认三栏里。

## 九点五、旧界面保留策略

第一阶段不要删除旧页面能力。

推荐：

```text
新入口：/workbench-v3 或 /workbench-dialogue
旧入口：/workbench-v2 保留
```

或者：

```text
默认进入新对话入口
提供“旧版工作台”入口用于回退
```

前端迁移时要保证：

```text
候选表仍可打开
状态环境仍可打开
图谱仍可打开
分支审稿仍可打开
```

新对话入口只是把它们组织得更顺手。

## 十、中文化

所有主界面中文优先。

禁止：

```text
乱码中文
用英文替代核心按钮
错误信息只有 HTTP 状态码
```

必须有中文：

```text
发送
停止
执行草案
取消草案
查看详情
刷新状态
切换场景
当前上下文
运行结果
等待确认
执行失败
```

技术字段可保留，但加中文标签：

```text
动作编号：action_id
迁移编号：transition_id
候选编号：candidate_item_id
任务编号：task_id
小说编号：story_id
```

## 十一、可参考实现

建议参考：

```text
assistant-ui：对话组件、线程、composer、流式状态。
CopilotKit：应用状态暴露、动作绑定、AI 助手融入业务页面。
LibreChat：线程列表、工具结果、Artifacts、多模型组织。
```

不建议：

```text
直接整体搬 Open WebUI 或 LibreChat。
把对话做成与状态系统无关的普通聊天页。
```

## 十二、前端第一阶段交付

第一阶段只做：

```text
对话主界面
任务场景切换
上下文摘要
分析任务草案卡片
审计草案卡片
运行卡片
结果 artifact
候选辅助页入口
状态辅助页入口
图谱辅助页入口
```

验收：

```text
作者打开网页后，看到的是对话线程和输入框。
作者能通过对话发起分析任务草案。
作者能看到分析执行过程。
作者能切换到审计上下文。
作者能通过对话生成审计草案。
作者能执行审计草案。
执行结果回到对话线程。
候选表和图谱不再抢占主界面。
```

额外验收：

```text
旧候选审计视图可以从对话打开。
旧状态环境视图可以从对话打开。
旧图谱视图可以从对话打开。
对话执行产生的 transition 能在图谱中高亮。
用户能从结果卡片跳到受影响的状态对象和分支。
```
