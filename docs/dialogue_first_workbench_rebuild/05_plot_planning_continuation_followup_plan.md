# 对话式剧情规划、续写与分支回流补充落地计划

本文档用于补齐“分析 -> 审计”之后的后半条真实作者链路：

```text
剧情规划
  -> 续写任务
  -> 生成分支
  -> 分支审稿
  -> 修改/重写/接受/丢弃
  -> 状态回流
```

结论先行：当前前后端第一阶段已经符合“状态机 + 对话主入口 + 动作草案 + 作者确认”的总体理念，但真实写小说工作流还需要继续把剧情规划和续写做成一等对话场景。后端已经提供工具入口，前端已有 artifact 和分支工作区基础，但还缺少明确的剧情规划 UX、续写上下文预览、生成后状态回流审计和端到端真实测试。

## 二、当前还没完全做到的部分

### 二点一、剧情规划还不是一等场景

后端已有 `create_plot_plan`，但前端报告中主要建议第一轮先测审计闭环。剧情规划还缺少明确的一等界面体验。

需要补齐：

```text
剧情规划场景入口
剧情规划上下文摘要
规划草案卡片
规划差异展示
规划确认后写入任务/状态环境
规划结果被续写任务引用
```

### 二点二、续写仍偏工具调用，不够像 Codex 式运行

后端已有 `create_generation_job`，但前端需要把续写过程展示成完整运行流：

```text
正在构建续写上下文
正在读取主状态
正在读取作者规划
正在检索参考证据
正在生成章节蓝图
正在生成草稿分支
正在保存分支 artifact
等待作者审稿
```

### 二点三、续写上下文预览不足

续写前作者必须能看到模型将使用什么：

```text
已确认角色状态
世界规则
剧情规划
禁止破坏项
伏笔
风格样例
RAG 证据
参考文本使用范围
上下文预算
```

这不是让作者读完整 JSON，而是给出压缩摘要和可展开详情。

### 二点四、分支审稿与状态回流还需强化

分支接受不是简单“接受文本”，而是：

```text
接受正文分支
分析新正文产生状态变化
生成状态候选
作者审计状态回流
确认后写入主状态
```

当前报告里有分支接受和 branch_graph 刷新，但还需要明确“生成后再分析回流”的闭环。

### 二点五、真实测试文档还缺后半链路

现有真实联调建议以审计闭环优先。需要新增后半链路测试：

```text
审计完成一批低风险状态
对话规划下一章
确认剧情规划
对话创建续写任务
生成分支
审稿分支
要求模型修改
接受或丢弃分支
生成后状态回流审计
```

## 三、目标工作流

作者在对话主入口中继续之前的分析审计场景。

### 三点一、剧情规划

作者输入：

```text
基于当前已确认状态，帮我规划下一章。重点推进主线矛盾，不要破坏已确认设定。给我三种剧情走向，不要生成正文。
```

模型返回：

```text
剧情规划草案 A：保守推进
剧情规划草案 B：强化冲突
剧情规划草案 C：铺垫伏笔
```

每个草案包含：

```text
章节目标
场景顺序
出场角色
角色状态变化
世界规则使用
伏笔推进
禁止破坏项
预计后续分支
风险提示
```

作者可以：

```text
执行草案
让模型合并 A 和 C
修改某个场景
保留为备选规划
取消
```

确认后：

```text
规划 artifact 写入对话线程
规划记录写入任务状态
续写上下文可以引用该规划
```

### 三点二、续写

作者输入：

```text
按照刚才确认的规划，生成下一章草稿分支。先生成一个分支，保持状态一致，不要破坏作者锁定字段。
```

模型生成动作草案：

```text
续写任务草案
  使用规划：plot_plan_xxx
  使用状态版本：state_version_no
  分支数量：1
  字数范围：作者指定或默认
  上下文预算：默认或作者指定
  参考证据范围：主故事 + 风格参考，不允许参考文本覆盖主线状态
```

作者确认：

```text
确认执行
```

系统执行并展示运行卡片：

```text
构建续写上下文
生成章节蓝图
生成正文
保存分支
生成分支 artifact
```

### 三点三、分支审稿

作者输入：

```text
审稿这个分支，检查它是否符合当前状态、规划和风格。列出问题，不要直接入库。
```

模型生成：

```text
分支审稿报告
一致性评分
风格匹配
状态破坏风险
需要重写的段落
建议动作：接受 / 修改 / 重写 / 丢弃
```

作者可以：

```text
让模型重写分支
接受分支入主线
丢弃分支
保留为备选
```

### 三点四、生成后状态回流

如果作者接受分支，系统不能只保存正文，还要进入状态回流：

```text
新章节正文
  -> 分析新文本
  -> 生成新增/变化状态候选
  -> 作者审计候选
  -> 写入主状态
```

这一步可以自动生成“状态回流审计草案”，但仍要作者确认。

## 六、真实测试链路

在你当前 123 真实任务上，建议下一轮按这个顺序：

```text
1. 打开 /workbench-v2/workbench-dialogue/
2. 选择 story_123_series_realrun_20260510
3. 选择 task_123_series_realrun_20260510
4. 切换到候选审计，接受一批低风险设定
5. 切换到剧情规划
6. 输入：基于当前状态生成三种下一章剧情规划，不写正文
7. 查看三份规划草案
8. 选择一份或要求模型合并
9. 确认执行规划草案
10. 切换到续写
11. 点击预览续写上下文

13. 执行生成一个分支
14. 查看分支 artifact
15. 切换分支审稿

17. 选择重写或接受/拒绝
18. 如果接受，触发生成后状态回流审计
19. 审计回流候选
20. 查看状态图、迁移图、分支图
```

## 七、验收标准

最低通过：

```text
剧情规划能通过对话生成草案。
规划草案能确认并保存 artifact。
续写能引用已确认规划。
续写前能预览上下文摘要。
续写执行能产生分支 artifact。
分支审稿能通过对话生成报告。
分支接受要求“确认入库”。
接受后能刷新 branch_graph。
```

推荐通过：

```text
生成后能自动创建状态回流审计草案。
状态回流候选能进入候选审计。
状态迁移图能展示新章节导致的状态变化。
分支图能展示生成、重写、接受、丢弃路径。
```

## 九、文档结论

当前系统已经符合“对话主入口 + 状态机核心”的主要方向，但真实写作链路还没有完全闭合。下一步要把剧情规划、续写、分支审稿和生成后状态回流做成与审计同等级的一等对话场景。

最终目标：

```text
作者在同一个对话式工作台里完成：
分析 -> 审计 -> 规划 -> 续写 -> 审稿 -> 回流状态 -> 再规划。
```

这才是完整的小说状态机写作环境。
