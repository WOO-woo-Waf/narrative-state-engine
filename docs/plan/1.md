
套餐

长篇小说全文分析与章节续写状态机改造计划
Summary
目标是把当前“先分析再续写”的骨架，升级成一套真正可处理 300K 级长文本的分层状态机流程：先按章节为主单位完成全文分析，再把 chunk 级结果归并为章节状态、全书状态和可复用梗概状态，后续续写始终基于这份完整状态运行，而不是只依赖一次性的 prompt 或局部摘要。

本次改造默认采用你确认的方案：

分析粒度：章节主导，chunk 仅作为章节内分析手段
持久化：双轨持久化，同时保留结构化仓储与完整 JSON 快照
Key Changes
1. 建立分层分析状态机，而不是只产出轻量分析资产
把当前 analysis 从“snippet/case/story_bible 的集合”升级为完整分层状态：

ChunkAnalysisState

summary
key_events
open_questions
character_mentions
world_rule_candidates
plot_thread_candidates
style_features
snippet_ids
coverage_flags
新增 ChapterAnalysisState
作为章节级主状态，字段固定为：

chapter_index
chapter_title
source_start_offset/source_end_offset
chunk_ids
chapter_summary
plot_progress
chapter_events
characters_involved
character_state_updates

新增 GlobalStoryAnalysisState
作为全文分析后的统一状态，字段固定为：

story_id/title

character_registry
plot_threads
world_rules
timeline_state
continuity_constraints

这样分析结果本身就是“可记录、可复用、可继续演化”的状态，而不是仅供一次续写消费的中间产物。

2. 分析流程改为“四段式”：切分、块分析、章节归并、全书归并
NovelTextAnalyzer 改为固定流水线：

章节切分
以章节标题为主；无标题时按长度生成伪章节，但对外仍保持章节编号。
章节内 chunk 切分
保证全文覆盖率，不能只分析前 N 句或前 N 个 snippet。
chunk 级抽取
针对每个 chunk 抽取事件、角色状态、世界规则线索、风格片段、未解问题。
章节归并
把同章多个 chunk 合并成一个 ChapterAnalysisState。
全书归并
跨章节去重、别名合并、剧情线串联、角色状态累计，形成 GlobalStoryAnalysisState。
梗概生成
先生成每章 chapter_synopsis，再由章节状态汇总出 story_synopsis。
这里的梗概定义为“状态摘要”，不是另写一篇自由文本简介。
必须同步修复当前分析器的三个核心缺口：

不能再只对 sentences[:max_snippets] 做截断式采样
chunk_states / story_synopsis / coverage / analysis_state 必须真实产出
character/world/plot/style 不能直接从全量原文一次扫完，必须经过“chunk -> chapter -> global”归并路径
3. 统一“分析状态”和“续写状态”的衔接方式
NovelAgentState 的职责重新划分为：

story
保存续写时要直接消费的 canonical story state
analysis
保存分析基线、章节索引、覆盖率、梗概和可回放的分析快照
具体调整：

扩展 AnalysisState：

analysis_version
baseline_global_state
chapter_states
chapter_synopsis_index
story_synopsis
coverage
retrieved_snippet_ids
retrieved_case_ids
snippet_bank
event_style_cases
evidence_pack
bootstrap.apply_analysis_to_state(...)
不再只映射 style/world/character/plot 的轻量字段，还要：

把 global_story_state 映射为续写起始 canonical state
把 chapter_states 写入 state.analysis.chapter_states
把 story_synopsis 写入 state.analysis.story_synopsis
写入 chapter_index -> summary/open_questions/anchors 索引，供后续章节续写检索
原则固定为：

分析期建立“基线状态”
每次续写产生“增量状态”
增量状态要能回写并更新全书状态和梗概状态
4. 建立章节续写状态机，而不是只在已有状态上循环生成
章节续写流程明确改为：

加载分析基线
从 JSON 或 repository 读取完整分析状态，而不是重新从 TXT 尾部摘要构造初始状态。
选择目标章节
续写默认以“下一章”为目标，也支持显式指定章节号。
组装章节续写上下文
固定包含：
global_story_state
最近若干章 chapter_states
与目标章节最相关的角色状态
未闭合 plot threads
世界规则与连续性约束
对应风格证据包
章节规划
新增章节级 planning 状态：
target_chapter_goal
required_plot_threads
required_character_arcs
continuity_must_keep
chapter_outline
多轮生成与校验
继续复用现有 continue_chapter_from_state(...) 框架，但输入改为完整分析状态驱动。
章节提交后反向更新
生成的新章节要再走一次同 schema 的章节分析，产出新的 ChapterAnalysisState，并：
更新 story.event_log
更新角色状态与剧情线状态
更新 analysis.chapter_states
更新 analysis.story_synopsis
形成新的 final.state.json
这样后续“第二次续写、第三次续写”都建立在已扩展的完整状态上。

Assumptions
章节是对外的基础单位，chunk 只是章节内部的分析实现细节。
第一版不强制引入场景级持久化；若章节内缺少明显场景边界，只保留 scene_markers 作为轻量字段。
梗概定义为“分析状态的摘要表达”，必须来源于结构化状态，不单独维护一套自由文本逻辑。
续写必须优先基于完整分析基线启动，不再默认走“只读原文尾部摘要”的弱基线方案。
保留现有 evidence retrieval、validation、repair loop、chapter completion policy；本次重点是把它们接到真正完整的全文分析状态上。
